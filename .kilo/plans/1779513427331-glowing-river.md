# 全量代码审计计划 — 玲音日记 (lingyin-webapp)

## 审计策略

本计划定义了审计的执行步骤和范围。鉴于项目处于 Phase 1 MVP 阶段（~78 个核心文件），采用**全量阅读 + 逐模块追踪调用链**的方式，对每一个文件进行安全审查。

审计遵循以下核心原则：
1. 数据库层（Prisma schema、migrations、`db.ts`中的查询逻辑）不在审计范围
2. 每个发现必须基于实际代码，必须追踪完整调用链
3. 如果无法确定某个问题是否可被利用，标记为"需进一步确认"
4. 审计完成后输出正式报告

## 发现列表

### 🔴 Critical: 0 个

### 🟠 High: 5 个

#### H1: API Key 明文写入 localStorage 导致泄露风险
- **文件**: `src/app/settings/page.tsx:68-71`
- **调用链**: Settings page → `handleSave()` → `localStorage.setItem(STORAGE_KEY, JSON.stringify({ provider, apiKey: draftApiKey }))`
- **问题**: 用户输入且即将提交到服务端的 API Key 被原封不动存入 `localStorage`。虽然项目声明"API Key 加密存储在服务器端"，但本地存储了一份明文副本，可被 XSS 攻击、恶意浏览器扩展或本地文件访问窃取。
- **影响**: 如果攻击者通过 XSS 或恶意扩展访问 localStorage，可窃取用户的 OpenRouter API Key，导致用户 API 额度被盗用。
- **修复建议**: 移除 `localStorage.setItem` 调用。API Key 的唯一真实来源应是服务端的加密存储。

#### H2: 所有 API 路由无速率限制
- **文件**: 所有 `src/app/api/**/route.ts`
- **问题**: 登录、注册、密码重置、邮件重发、AI 生成等任何端点均无速率限制。攻击者可：
  - 暴力破解密码（`/api/auth/callback/credentials` 不经过自定义 API route，但登录尝试本身无限制）
  - 轰炸式发送验证邮件/密码重置邮件（消耗 Resend 额度）
  - 滥用 AI 生成消耗用户的 API 额度
  - DDoS 攻击所有端点
- **涉及端点**: `/api/auth/register`, `/api/auth/resend-verification`, `/api/auth/forgot-password`, `/api/entries`, `/api/ai/generate`, `/api/ai/rewrite`, `/api/ai/test` 等
- **修复建议**: 使用 `@upstash/ratelimit` + `@upstash/redis`（Vercel Edge 兼容）或 `express-rate-limit` 风格的中间件实现基于 IP + userId 的速率限制。

#### H3: `/api/health` 端点泄露基础设施信息
- **文件**: `src/app/api/health/route.ts:1-48`
- **调用链**: GET `/api/health` → 无鉴权 → 返回所有环境变量状态、服务可用性
- **问题**: 该端点完全公开（middleware 明确放行 `/api/*`），返回内容包括哪些环境变量已配置、数据库连接状态等基础设施细节。攻击者可利用这些信息进行针对性攻击。
- **影响**: 信息泄露，帮助攻击者绘制基础设施地图。
- **修复建议**: 
  1. 对 health 端点添加简单的鉴权（如共享 secret token）
  2. 移除 `checks.env` 部分的环境变量细节输出
  3. 仅返回 HTTP 200/500 状态码，不返回任何细节

#### H4: AI Test 端点可被用作 OpenRouter 代理
- **文件**: `src/app/api/ai/test/route.ts:25-121`
- **调用链**: POST `/api/ai/test` → 鉴权检查 → 接收请求体中的 `apiKey` → 直接使用该 key 调用 OpenRouter API
- **问题**: `/api/ai/test` 接受请求体中的 `apiKey` 参数并直接用其发起 API 调用。虽然该端点要求用户已登录，但攻击者可在自己的账户下使用自己的 API Key 测试（正常行为），也可使用窃取的 key 进行测试。更关键的是，**Preflight 检查**（第55-66行）对 openrouter.ai 发起无认证的 GET 请求，该请求使用 `AbortSignal.timeout(10000)` 和 `fetch`，如果传入可控 URL 则存在 SSRF 风险。当前 URL 是硬编码的，但缺乏警惕。
- **修复建议**: 
  1. 移除 preflight 网络连通性检查（或仅在生产环境启用且使用固定超时）
  2. 确保 `apiKey` 参数仅用于测试用途，添加明确的审计日志
  3. 对该端点实施严格的速率限制

#### H5: 错误信息泄露至客户端（AI 生成端点）
- **文件**: `src/app/api/ai/generate/route.ts:63-68`, `src/app/api/ai/rewrite/route.ts:59-64`
- **调用链**: AI 生成失败 → `catch(error)` → 将 `error.message` 直接通过 SSE 发送给客户端
- **问题**: 当 AI 生成失败时，原始错误消息（可能包含 API endpoint、模型名、部分请求内容）被直接发送给客户端。在 `src/app/api/ai/test/route.ts:9-23` 的 `logConnectionError` 中，完整的错误堆栈被 `console.error` 输出，可能被 Vercel Logs 捕获。
- **影响**: 敏感信息（如 API endpoint、网络拓扑）可能泄露。
- **修复建议**: 
  1. SSE error 消息应使用通用错误提示（如 "生成失败，请稍后再试"）
  2. 仅在 `process.env.NODE_ENV === "development"` 时返回详细错误

### 🟡 Medium: 10 个

#### M1: 缺少 CSP 及其他安全响应头
- **文件**: `next.config.mjs` — 未配置 `headers()`
- **问题**: 没有设置 Content-Security-Policy、X-Content-Type-Options、X-Frame-Options、Strict-Transport-Security 等安全响应头。
- **修复建议**: 在 `next.config.mjs` 中添加 `headers()` 配置，或使用 Vercel 的 `vercel.json` 中的 `headers` 字段。

#### M2: 缺少输入 Schema 校验库
- **范围**: 所有 API routes、Server Actions
- **问题**: AGENTS.md 提到"推荐使用 Zod 做服务端校验"，但 `package.json` 中没有安装任何 schema 校验库。所有 API 端点都依赖手动类型断言和简单的 if-check。这增加了类型安全风险，尤其是当参数数量增加时。
- **修复建议**: 安装 `zod`，为每个 API 端点的请求体定义 Zod schema 并验证。

#### M3: 邮件重发和密码重置端点存在用户枚举风险
- **文件**: `src/app/api/auth/resend-verification/route.ts`, `src/app/api/auth/forgot-password/route.ts`
- **调用链**: POST 端点 → `resendVerification(email)` / `forgotPassword(email)` → 查询用户是否存在 → 返回不同响应时间
- **问题**: 虽然业务逻辑对不存在用户返回 `ok:true`（防止直接枚举），但 Prisma 查询的响应时间差异可被时序攻击利用来判断用户是否存在。这是一个已知的基于时间的用户枚举向量。
- **修复建议**: 对所有请求添加固定延迟（如 500ms）或使用相同的处理路径。

#### M4: 文件上传 MIME 类型信任客户端输入
- **文件**: `src/app/api/upload/route.ts:36`
- **问题**: `file.type` 由浏览器提供，可被伪造。恶意用户可上传一个内容为 `text/html` 的文件但声明 `Content-Type: image/png`，虽然代码使用 `file.type` 做检查（而非检查文件内容），但实际影响有限，因为：
  1. 文件存储到 R2，不是公开目录
  2. 通过 presigned URL 访问
  3. 客户端使用 `next/image` 渲染
- **风险**: 低。如果将来图片直接嵌入 HTML 或以其他形式渲染，可能引发问题。
- **修复建议**: 使用 `file-type` 或 `magic-bytes` 库对上传文件进行服务端魔术数字检测。

#### M5: 文件名生成使用非密码学安全的随机数
- **文件**: `src/app/api/upload/route.ts:16`
- **问题**: `Math.random()` 用于生成文件名中的随机部分，这不可预测但不满足密码学要求。攻击者可预测文件名（如果知道时间戳），但数据通过 R2 presigned URL 和 userId 路径隔离，实际利用难度高。
- **修复建议**: 使用 `crypto.randomUUID()` 替代 `Math.random()` 生成文件名（项目已在 `auth-service.ts:6` 使用 `crypto.randomUUID()`）。

#### M6: expand_env.py 硬编码绝对路径
- **文件**: `scripts/expand_env.py:5`
- **问题**: `ENV_PATH = '/Users/sunrunchen/GitHub/lingyin-webapp/.env'` 硬编码了开发者本地路径。这个脚本在任何其他机器上运行都会失败。更重要的是，该脚本写入 `.env` 文件，如果他人运行将覆盖自己的配置。
- **修复建议**: 改为相对路径 `os.path.join(os.path.dirname(__file__), '..', '.env')` 或使用环境变量。

#### M7: 公开 demo.html 使用 CDN 加载 Tailwind
- **文件**: `public/demo.html:7`
- **问题**: 生产环境中包含一个从 `cdn.tailwindcss.com` 加载的 demo 页面。虽然该页面本身不包含敏感信息，但：
  1. 依赖外部 CDN 的可用性
  2. 如果 CDN 被劫持，可能执行恶意脚本（高度理论性，因为 Tailwind CSS CDN 是可信的）
- **修复建议**: 将 demo.html 移出 `public/` 目录，或将其标记为开发用途。

#### M8: Password Reset Token 在 URL 中传递
- **文件**: `src/lib/email.ts:47` — `const resetUrl = \`${baseUrl}/reset-password?token=${token}\``
- **问题**: 密码重置令牌通过 URL 查询参数传递，可能被浏览器历史、Referer 头、服务器日志记录。
- **修复建议**: 理想方案是使用 POST 请求携带 token，但这对邮件链接不友好。当前实现是业界常见做法（大多数网站如此），风险可接受。但应确保：
  1. Token 的过期时间足够短（当前是 1 小时 ✓）
  2. 使用后立即失效（当前实现 ✓）

#### M9: AppShell 客户端 Auth 检查使用 fetch 但无错误处理
- **文件**: `src/components/AppShell.tsx:24-34`
- **问题**: `fetch("/api/auth/session")` 调用没有在非 200 响应时的显式处理，且 `catch` 块仅设置了 `setAuthLoaded(true)` 但不清除 auth 状态。如果 session API 返回错误，`authenticated` 保持为 SSR 传入的初始值（可能过期）。
- **修复建议**: 在 catch 块中设置 `setAuthenticated(false)`。

#### M10: Service Worker 缓存日记 API 响应
- **文件**: `src/sw.ts:30-40`
- **问题**: `/api/entries` 的 NetworkFirst 策略缓存日记元数据（preview、tags 等），最大 50 条目、1 小时过期。若用户在公共设备上登录后缓存了日记列表，退出登录后其他用户可能从 Service Worker 缓存中看到日记预览。
- **风险**: 中等。需要物理设备访问才能利用。
- **修复建议**: 在用户登出时清除 Service Worker 缓存，或降低缓存 maxAgeSeconds。

## 总体评估

### 安全评级: B+ (良好)

项目整体安全态势良好，尤其是对于 Phase 1 MVP。核心安全机制（加密、鉴权、权限隔离）实现得当。主要短板在于：
1. API Key 在 localStorage 的副本
2. 无速率限制
3. Health 端点过于公开
4. 无测试覆盖

### 是否适合上线

**可以上线**，但建议在上线前修复 H1、H2、H3。

理由：项目没有可直接导致数据泄露或系统失陷的 Critical 漏洞。影响最大的问题（localStorage API Key 副本、无速率限制）可以通过快速修复解决。Health 端点的信息泄露是无心之失，修复简单。

### Top 10 优先修复项

| # | 问题 | 严重度 | 建议 |
|---|------|--------|------|
| 1 | H1 - API Key 写入 localStorage | High | 删除 `localStorage.setItem` 调用 |
| 2 | H2 - 无速率限制 | High | 添加 Upstash 速率限制 |
| 3 | H3 - Health 端点信息泄露 | High | 移除环境变量详情输出 |
| 4 | M1 - 缺少 Security Headers | Medium | 在 next.config 添加 headers |
| 5 | H4 - AI Test 端点代理风险 | High | 添加速率限制 + 移除 preflight |
| 6 | M2 - 缺少输入 Schema 校验 | Medium | 安装 Zod 并逐步添加 schema |
| 7 | M9 - AppShell auth 状态管理 | Medium | 改进错误处理 |
| 8 | L2 - 无自动化测试 | Low | 为核心路径添加测试 |
| 9 | M10 - SW 缓存日记数据 | Medium | 登出时清理缓存 |
| 10 | H5 - AI 错误信息泄露 | High | 客户端返回通用错误消息 |

### 后续安全加固建议

1. **上线后**: 
   - 启用 Vercel 的 WAF 和 DDoS 保护
   - 配置 CSP 和 HSTS 头
   - 监控 Vercel Logs 中是否有异常访问模式

2. **Phase 2 开发前**:
   - 安装 Zod 并统一所有 API 输入校验
   - 为核心安全路径添加集成测试
   - 添加 CI/CD pipeline
   - 实施速率限制

3. **持续改进**:
   - 考虑使用 CSP report-only 模式收集违规报告
   - 定期运行 `npm audit` 检查依赖漏洞
   - 考虑引入 `helmet` 或类似的安全头中间件
