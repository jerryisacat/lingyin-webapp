# 前端全量代码重构与 UI/UX 升级方案 (Issue #96 & UI 升级)

## 1. 目标与范围

本次重构的目标是解决 AI-driven 开发带来的「一致性熵增」，统一前端代码的规范和架构，同时**进行全站的 UI/UX 体验重构**。产品定位为“温暖、治愈的个人日记空间”，我们将在现有的樱花粉/暖白调性基础上，从排版、响应式布局、设计 Tokens 等方面进行现代化的视觉提升。

## 2. 设计规范与 UI/UX 重构 (New Design System)

### 2.1 视觉调性与配色 (Visual Tone & Colors)
*   **设计语言:** 温暖治愈风 (Warm & Cozy)。强调呼吸感（大圆角、充裕的 padding）、柔和的阴影和微交互动画。避免大面积高对比度的刺眼色彩。
*   **Color Tokens 重构:** 在 `tailwind.config.ts` 中规范化色板，并强制推行。
    *   `bg-background`: 主背景（暖白 `warm-white`）。
    *   `bg-surface`: 卡片/模块背景（如纯白带软阴影，或浅米色）。
    *   `text-primary` / `text-secondary` / `text-muted`: 替换现有的 `text-ink` / `text-ink-light`，建立严格的文本层级。
    *   `brand-primary` / `brand-light`: 统一现有的 Sakura 体系。

### 2.2 响应式布局架构 (Responsive Layout Architecture)
彻底重构现有的 `AppShell` 结构，提供真正区分桌面/移动端的体验：
*   **Desktop (桌面端 / md 以上):**
    *   采用 **侧边栏布局 (Sidebar)**。左侧固定导航栏（Logo、写日记、时间轴、数据统计、设置），右侧为宽敞的内容/工作区。
    *   利用大屏幕空间，时间轴可以采用更舒展的瀑布流或网格形式，编辑器提供类似 Notion 的沉浸式阅读/书写体验。
*   **Mobile (移动端 / sm 以下):**
    *   顶部保留简洁的 Header。
    *   底部保持 `MobileTabBar`，确保单手操作友好。
    *   卡片横向铺满，增加防触误距 (safe-area-inset)。

### 2.3 核心交互与微动效提升
*   **打字机与 AI 响应:** 优化流式输出时的平滑度，防止页面跳动。
*   **日历与卡片:** 引入悬浮态 (Hover state) 微放大、点击涟漪等交互反馈。
*   **转场动画:** 页面切换时引入 Framer Motion 的渐隐渐出或平滑滑动效果。

## 3. 代码架构治理任务

### 3.1 类型体系清理 (TypeScript)
*   **清理淘汰类型:** 删除 `src/types/index.ts` 中不再使用的 `Tone` 相关类型及其残留的引用，完全切换到 `WritingStyle` (Persona + Perspective)。
*   **清理冗余 Prompt:** 删除 `src/lib/ai/prompts.ts` 中为了向下兼容而保留的 `WARM_SYSTEM_PROMPT` 等遗留常量（如果已不再需要）。
*   **Props 接口规范:**
    *   统一所有组件使用 `interface ComponentNameProps { ... }` 格式。
    *   将 `type ComponentNameProps = ...` 或内联定义的 Props 全部重构为 `interface`。
    *   **位置策略:** 所有 Props 定义与组件保持在**同一文件中**，保持高内聚。

### 3.2 组件架构与规范
*   **统一导出风格:**
    *   页面组件 (`src/app/**/page.tsx`): 统一使用 `export default function PageName() { ... }`。
    *   共享组件 (`src/components/**`): 统一使用命名导出 `export function ComponentName() { ... }`，便于代码重构和导入提示。
*   **大组件拆分:**
    *   重构 `DiaryEditor.tsx` (294行)，按职责拆分为：
        *   `DiaryEditorToolbar`: 顶部操作区、状态提示、返回按钮。
        *   `DiaryEditorInput`: 文本输入区、写作风格配置(Persona/Perspective)选择器。
        *   `DiaryEditorPreview`: 包含 Markdown 渲染、打字机效果及图片渲染。
    *   重构 `WritingStyleConfig.tsx` (如适用)：将其内部的多个 step 拆分为子组件（如果逻辑过长）。
*   **清理无用代码:** 删除未使用的组件、未使用的导入(Imports)和无用状态。

### 3.3 状态管理层 (Global State)
*   **策略:** 采用 **React Context** 处理跨组件共享数据（避免引入外部依赖）。
*   **新增 Context:**
    *   创建 `src/contexts/UserConfigContext.tsx` 或类似的文件，用于全局管理和缓存用户的 `WritingStyle` (Persona) 以及其他可能的全局偏好设置。
    *   移除各组件（如 `DiaryEditor`）内部独立的 `fetch("/api/user/style")` 逻辑，统一通过 `useUserConfig()` Hook 获取。

### 3.4 API 层统一
*   **响应格式标准化:**
    *   新建 `src/lib/api-response.ts`，将现有的 `jsonOk` 和 `jsonError` 集中管理。
    *   将 `src/app/api/` 下所有路由的 `NextResponse.json(...)` 统一替换为 `jsonOk` / `jsonError`。
*   **客户端调用规范:** 梳理前端的 `fetch` 调用，统一处理 `{ ok, data, error }` 格式。

### 3.5 样式代码治理
*   清理 `globals.css` 中的陈旧 class，全面拥抱 Tailwind Utility Classes。
*   如果有复杂的组件特定样式，尽量提取到单独的 css module 或通过 Tailwind `@layer components` 规范化，而不是写内联 `style={{...}}`。

## 4. 实施阶段划分

1.  **Phase 1: 基础清理与类型统一** (清理死代码、重构 Props 接口、定义全局 Context)。
2.  **Phase 2: UI 框架与布局重构** (修改 tailwind 配置，重写 `AppShell` 实现桌面侧边栏与移动端底栏的双轨响应式)。
3.  **Phase 3: 核心页面重绘** (重写 DiaryEditor、时间轴、设置页，应用新的设计 Tokens)。
4.  **Phase 4: API 统一与收尾** (重构 API Route 响应格式，确保 tsc 和 lint 零报错)。
