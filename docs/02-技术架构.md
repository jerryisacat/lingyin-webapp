## 2. 技术栈选择

### 2.1 核心框架

| 技术 | 选择 | 理由 |
|------|------|------|
| **框架** | Next.js 14+ (App Router) | 全栈，前后端统一，PWA 友好 |
| **语言** | TypeScript | 类型安全，协作友好 |
| **样式** | Tailwind CSS | 快速开发，响应式内置 |
| **PWA** | Serwist (ex-next-pwa) | Next.js 原生 PWA 支持 |
| **数据库** | Supabase PostgreSQL | 云端托管，Phase 1 直接上 PG |
| **ORM** | Prisma (datasource: postgresql) | 类型安全，迁移方便 |
| **认证** | Supabase Auth (Magic Link) | 内置，无需 Auth.js |
| **文件存储** | CloudFlare R2 (S3 兼容) | 零出站费，S3 API，全局边缘 |

### 2.2 Supabase Auth Session 管理

使用 `@supabase/ssr` 包在 Next.js App Router 中管理 session：

| 层 | 方法 | 说明 |
|----|------|------|
| **服务端** | `createServerClient` | Route Handlers / Server Components 中读取 cookie 中的 session |
| **客户端** | `createBrowserClient` | 浏览器组件中自动管理 cookie |
| **中间件** | `middleware.ts` + `updateSession` | 拦截所有路由，验证 session，未登录重定向 `/login` |

Auth 流程:
```
用户访问 → middleware 检查 session cookie → 未登录 → 重定向 /login
用户输入邮箱 → Supabase Auth 发送 Magic Link
用户点击链接 → @supabase/ssr 设置 session cookie → 重定向首页
后续请求 → 自动携带 cookie → middleware 验证通过
```

### 2.3 AI / LLM

| 技术 | 说明 |
|------|------|
| **LLM SDK** | `openai` npm 包（兼容 OpenAI / DeepSeek / Groq 等） |
| **流式输出** | Server-Sent Events (SSE)，打字机效果 |
| **Prompt 模板** | 每个语气对应一套 System Prompt，可扩展 |
| **图片分析** | Gemini Vision / GPT-4o / Qwen-VL 多模态 |
| **用户 API Key** | 前端 localStorage 存储，请求时附带 |

### 2.4 部署

| 环境 | 选型 | 说明 |
|------|------|------|
| **Phase 1 部署** | Vercel (with Supabase Integration) | 免费层够用，自动 SSL |
| **域名** | `lingyin.app` 或类似 | Vercel 自动配置 SSL |
| **监控** | Sentry (免费) + Vercel Analytics | 错误追踪 + 性能监控 |
| **CI/CD** | Vercel Git Integration | push → auto deploy |

### 2.5 前后端分离评估 (已否决 MVP)

| 方案 | 前端 | 后端 API | 结论 |
|------|------|----------|------|
| **A (当前)** | Vercel | Vercel (同仓) | ✅ MVP 采用 |
| **B** | CF Pages | Vercel (跨域) | ❌ Auth cookie 跨域 + CORS 复杂度 |
| **C** | CF Pages | CF Workers | ❌ Workers 不支持 Prisma TCP，需重写后端 |

**拒绝方案 B/C 的核心原因:**
1. Supabase Auth session cookie 绑定在 API 域名，前端跨域无法读取，需改为 token-based auth
2. CORS 预检请求每个 API 调用增加 100-200ms 延迟
3. 双份部署配置 + PR 预览环境双倍管理成本
4. CF Workers 无原生 TCP 连接，Prisma 无法直接运行，需 Prisma Accelerate（额外 $20/月）或改用 D1

**结论:** Phase 1 沿用 Next.js 单体部署 Vercel。R2 作为存储与 Vercel 无冲突（服务端 S3 API 读写）。前后端分离推迟到 Phase 3+。

## 3. 数据流

### 3.1 AI 写日记流程

```
用户上传图片+文字
        │
        ▼
  [前端] 读取 localStorage 中的 API Key
        │
        ▼
  [前端] POST /api/ai/generate
        │  body: { text, images, tone }
        ▼
  [后端 aiService]
    ① 构建 Prompt（根据语气选择模板）
    ② 构建 User Message（文字+图片描述）
    ③ 调用 LLM API（流式，附带 X-API-Key）
        │
        ▼
  [后端 → 前端] SSE 流式返回生成结果
        │
        ▼
  [前端 DiaryEditor] 实时显示 AI 生成内容（打字机动画）
        │  用户可以立即修改
        ▼
  用户点击「保存」
        │
        ▼
  [后端] 保存 Markdown 到 R2 + 写入 Entry 元数据到 PostgreSQL
        │
        ▼
  保存成功，跳转日记详情页
```

### 3.2 存储结构 (CloudFlare R2)

日记文件和图片存储在 CloudFlare R2 中，按用户和日期组织：

```
{bucket}/users/{userId}/entries/2026/05/
  ├── 2026-05-19.md          ← 日记正文 (Markdown)
  └── assets/                ← 图片
      ├── IMG_20260519_001.jpg
      └── IMG_20260519_002.png
```

> **注意:** Vercel 无持久化文件系统，所有文件必须使用 CloudFlare R2。开发环境可在 `data/` 目录模拟本地文件系统，但 `data/` 需加入 `.gitignore`。<br>
> **R2 优势:** 零出站费（egress free），S3 兼容 API，全球边缘节点。使用 `@aws-sdk/client-s3` 读写。

数据库仅存元数据（详见 `docs/05-数据模型.md` 中的 Prisma Schema）。`Entry.preview` 字段保存 Markdown 前 200 字，时间线列表查询不走 R2。

## 4. PWA 策略

### 4.1 Service Worker 缓存策略

| 资源类型 | 策略 | 说明 |
|---------|------|------|
| **页面 Shell** | Cache First | 首页、设置页等 App Shell |
| **日记内容** | Network First | 优先从服务器获取，离线回退缓存 |
| **图片/视频** | Cache First | 已加载的多媒体文件本地缓存 |
| **API 请求** | Network Only | 写操作和 AI 生成需联网 |
| **字体/CSS/JS** | Stale While Revalidate | 构建资源秒级加载 |

## 5. API 设计

### 5.1 主要接口

| 方法 | 路径 | 说明 | 鉴权 |
|------|------|------|------|
| GET | /api/entries | 获取日记列表（分页） | Session |
| GET | /api/entries/:date | 获取某天日记 | Session |
| POST | /api/entries | 保存新日记 | Session |
| PUT | /api/entries/:id | 修改日记 | Session |
| DELETE | /api/entries/:id | 删除日记 | Session |
| POST | /api/ai/generate | AI 生成日记（流式 SSE） | Session + X-API-Key |
| POST | /api/ai/rewrite | AI 重写/润色 | Session + X-API-Key |
| POST | /api/upload | 上传图片 | Session |
| GET | /api/user/config | 获取用户配置 | Session |
| PUT | /api/user/config | 更新用户配置 | Session |

### 5.2 API Key 守卫

- 前端: 访问 `/diary` 页面前检查 `localStorage` 中是否有 `apiKey`
- 若缺失 → 重定向 `/settings`，提示 "请先配置 API Key"
- 后端: `/api/ai/*` 路由检查 `X-API-Key` 头，缺失返回 `401 { error: "API Key required" }`

## 6. 安全与隐私

### 6.1 API Key 处理

- **Phase 1:** 用户 API Key 存储在浏览器 localStorage
- **前端 → 后端:** 请求时通过 Header `X-API-Key` 传递
- **后端:** 透传到 LLM API，**不记日志、不持久化、不落库**
- **Phase 3:** API Key 由你的服务器管理，用户不接触

### 6.2 数据隐私

- 日记明文存储（Markdown），通过 R2 访问控制保护
- 用户 ID 使用 UUID，不暴露邮箱
- 所有传输走 HTTPS
- 可选的端到端加密（Phase 4 考虑）

### 6.3 Row Level Security (RLS) — Supabase PostgreSQL

所有 Supabase 表必须启用 RLS。**R2 不需要 RLS** —— 访问由服务端 S3 凭证控制。

**表级 RLS 策略:**

| 表 | 策略 | SQL |
|----|------|-----|
| **User** | 用户只能读写自己 | `(SELECT auth.uid()) = id` |
| **Entry** | 用户只能 CRUD 自己的日记 | `(SELECT auth.uid()) = "userId"` |
| **Subscription** | 用户只能读自己的订阅 | `(SELECT auth.uid()) = "userId"` |
| **Invoice** | 用户只能读自己的账单 | `(SELECT auth.uid()) = "userId"` |

**服务端操作:** API Route Handlers 使用 `SUPABASE_SERVICE_ROLE_KEY` 创建的 Supabase client 可绕过 RLS，用于跨用户的管理操作。

```sql
-- 启用 RLS
ALTER TABLE public."User" ENABLE ROW LEVEL SECURITY;
ALTER TABLE public."Entry" ENABLE ROW LEVEL SECURITY;

-- Entry 表策略示例
CREATE POLICY "Users can only access their own entries"
ON public."Entry"
USING (auth.uid() = "userId");
```

### 6.4 Prisma ↔ Supabase Auth 用户同步

Prisma `User.id` = Supabase Auth `auth.users.id`。

**推荐方案: Database Trigger（自动同步）**

```sql
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS trigger AS $$
BEGIN
  INSERT INTO public."User" (id, email) VALUES (new.id, new.email);
  RETURN new;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();
```

**备选方案:** 首次 API 请求时检查 Prisma 中是否存在该 User，不存在则创建（要求 API 有写权限）。

## 7. 环境变量参考

| 变量 | 用途 | 获取方式 | 使用位置 |
|------|------|----------|----------|
| `DATABASE_URL` | Prisma 连接 Supabase PG | Supabase Dashboard → Settings → Database → Connection string (port 5432) | 服务端 (Prisma) |
| `SUPABASE_URL` | Supabase 项目 API URL | Supabase Dashboard → Settings → API → Project URL | 服务端 |
| `SUPABASE_ANON_KEY` | 匿名公钥 (RLS 受限) | Supabase Dashboard → Settings → API → anon key | 客户端 + 服务端 |
| `SUPABASE_SERVICE_ROLE_KEY` | 服务端密钥 (绕过 RLS) | Supabase Dashboard → Settings → API → service_role key | **仅服务端** |
| `NEXT_PUBLIC_SUPABASE_URL` | 同 `SUPABASE_URL` (暴露浏览器) | 同上 | 客户端 |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | 同 `SUPABASE_ANON_KEY` (暴露浏览器) | 同上 | 客户端 |
| `R2_ACCESS_KEY_ID` | R2 S3 API 访问密钥 ID | CloudFlare Dashboard → R2 → Manage API Tokens | 服务端 |
| `R2_SECRET_ACCESS_KEY` | R2 S3 API 访问密钥 | 同上 | 服务端 |
| `R2_ENDPOINT` | R2 S3 API 端点 | CloudFlare Dashboard → R2 → Settings (如 `https://<id>.r2.cloudflarestorage.com`) | 服务端 |
| `R2_BUCKET` | R2 Bucket 名称 | 自行创建 | 服务端 |
| `R2_PUBLIC_URL` | R2 公开访问域名 | CloudFlare Dashboard → R2 → Settings → Public access | 服务端 (构造文件 URL) |
